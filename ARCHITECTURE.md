## Data Model (Firestore)

```
users/{uid}                         # doc created lazily on first message
  ├─ email: string | null
  ├─ displayName: string | null
  ├─ createdAt: timestamp
  └─ updatedAt: timestamp
  │
  └─ chats/{chatId}                 # chatId is UUID generated by CLIENT
       ├─ title: string             # first ~60 chars of first user message
       ├─ createdAt: timestamp
       ├─ updatedAt: timestamp      # bumped on every new message
       │
       └─ messages/{messageId}      # messageId auto-generated by Firestore
            ├─ role: "user" | "assistant"
            ├─ parts: Array<{ type: string, text?: string }>
            └─ createdAt: timestamp
```

**Path-based ownership:** Every doc lives under `users/{uid}/...`. The uid IS the ownership marker — no separate `owner` field needed. Security rules enforce `request.auth.uid == uid` at every layer.

**Client generates chatId:** UUIDs are generated in the browser (`crypto.randomUUID()`) before sending the first message. This lets us mirror the id in the URL immediately without a round-trip.

## Auth Flow

```
User visits `/`
        │
        ▼
`(app)/layout.tsx` (client component)
   useAuth() → loading, user
        │
        ├─ if loading:  → show "loading" spinner
        ├─ if !user:    → router.replace("/login")
        └─ if user:     → render Sidebar + children
                          │
                          ▼
                   ChatView → transport.prepareSendMessagesRequest
                          → await getIdToken()
                          → Authorization: Bearer <token>
                          │
                          ▼
                   POST /api/chat
                          → adminAuth.verifyIdToken(token)
                          → decoded.uid
```

**Client-side auth only.** No session cookies, no middleware auth guard. The layout redirects unauthenticated users to `/login`. The API route verifies every request via ID token in the Authorization header.

**ID token refresh** happens transparently — `auth.currentUser.getIdToken()` returns a fresh token if the cached one is close to expiring.

## Chat Request Lifecycle

```
1. User submits message in ChatView
        │
        ▼
2. useChat sends POST /api/chat with:
   {
     messages: [...history, newUserMessage],
     chatId: "abc-uuid-generated-client-side"
   }
   Headers: Authorization: Bearer <idToken>
        │
        ▼
3. /api/chat/route.ts:
   a. Verify auth token             (~50-100ms)
   b. Fire off 3 Firestore writes in parallel (fire-and-forget):
        - ensureUser(uid, ...)
        - ensureChat(uid, chatId, firstText)
        - appendMessage(uid, chatId, userMessage)
   c. retrieveContext(query, 6)     (~10-30ms — in-memory BM25)
   d. streamText({ model, system, messages, onFinish })
        │
        ▼
4. Stream flows back to client immediately (Server-Sent Events)
        │
        ▼
5. useChat appends chunks to messages state → UI renders live
        │
        ▼
6. Stream completes:
   a. onFinish (server-side): await pendingWrites; save assistant message; touchChat
   b. onFinish (client-side): if it's a brand-new chat, mirror id in URL
        │
        ▼
7. Sidebar's onSnapshot listener fires → new chat appears in list
```

**Parallelism matters:** We DON'T await Firestore writes before starting the Gemini stream — that would add ~400-1000ms of latency the user perceives as "the model is slow." The writes settle in the background; `onFinish` awaits them before saving the assistant reply.

## RAG Pipeline (BM25)

**Corpus:** 3 PDFs → cleaned text → chunked into 1,854 pieces (~800-1000 chars each) → saved to `lib/knowledge/chunks.json` (~2.9MB).

**At module load** (`lib/rag.ts`):
1. Import `chunks.json`
2. Tokenize each chunk (handle Hebrew prefixes ה/ו/ב/ל/כ/מ/ש)
3. Compute term frequencies (TF), document lengths, and document frequencies (DF)
4. Everything lives in memory as `Map<string, number>[]`

**At query time** (`retrieveContext(query, topK=6)`):
1. Tokenize query (same rules)
2. Score every chunk with BM25 (k1=1.5, b=0.75)
3. Sort descending, take top K
4. Format as `[מקור {i} — {sourceTitle}, קטע {idx}]\n{text}` blocks

**Why BM25 not embeddings?**
- Free-tier Gemini embedding quota is 100 requests/day — too low to build the index
- Hebrew tax terminology is highly specific ("מע"מ", "עוסק פטור", section numbers) — keyword match works well
- Zero external dependencies at query time

**Tokenizer:** Character-based (not regex-based) to avoid bundler mangling of Hebrew Unicode ranges. See `lib/rag.ts::tokenize()`.

## Security Rules (Firestore)

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{uid} {
      allow read: if request.auth != null && request.auth.uid == uid;
      allow write: if false;              // server-only via Admin SDK

      match /chats/{chatId} {
        allow read: if request.auth != null && request.auth.uid == uid;
        allow write: if false;

        match /messages/{messageId} {
          allow read: if request.auth != null && request.auth.uid == uid;
          allow write: if false;
        }
      }
    }
    match /{document=**} { allow read, write: if false; }
  }
}
```

**All writes deny by default.** Firebase Admin SDK bypasses these rules on the server. Clients can only read, never write. This makes the API route the single source of truth for state changes.

## Streaming — How It Actually Works

**Server side:**
- `streamText(...)` returns a `StreamTextResult` synchronously
- `.toUIMessageStreamResponse()` returns a `Response` with `Content-Type: text/event-stream`
- The body is a `ReadableStream` that emits SSE events as Gemini produces tokens
- `onFinish` callback fires when the stream completes (server-side)

**Client side:**
- `useChat` uses `DefaultChatTransport` to POST to `/api/chat`
- Reads the SSE stream chunk by chunk
- Each `text-delta` chunk appends to the current assistant message
- `onFinish` callback fires when the stream completes (client-side)

**Streaming events format** (from server):
```
data: {"type":"start"}
data: {"type":"text-start","id":"0"}
data: {"type":"text-delta","id":"0","delta":"התשובה"}
data: {"type":"text-delta","id":"0","delta":" מתחילה"}
...
data: {"type":"text-end","id":"0"}
data: {"type":"finish","finishReason":"stop"}
data: [DONE]
```

## Environment Variables

| Variable | Where used | Client or Server? |
|---|---|---|
| `GOOGLE_GENERATIVE_AI_API_KEY` | `@ai-sdk/google` reads it | Server-only |
| `NEXT_PUBLIC_FIREBASE_API_KEY` | `firebase-client.ts` | Client + Server |
| `NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN` | `firebase-client.ts` | Client + Server |
| `NEXT_PUBLIC_FIREBASE_PROJECT_ID` | `firebase-client.ts` | Client + Server |
| `NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET` | `firebase-client.ts` | Client + Server |
| `NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID` | `firebase-client.ts` | Client + Server |
| `NEXT_PUBLIC_FIREBASE_APP_ID` | `firebase-client.ts` | Client + Server |
| `FIREBASE_ADMIN_JSON` | `firebase-admin.ts` (prod fallback) | Server-only |

**`NEXT_PUBLIC_*` prefix:** Next.js bundles these into the client JS. They ARE visible in the browser. Firebase Web config is designed to be public — real security comes from Firestore rules + Firebase Auth, not from hiding these keys.

**`FIREBASE_ADMIN_JSON`:** Read only server-side. Contains the private key that bypasses all security rules. If leaked, the attacker has full DB access.

## Performance Characteristics

**Cold start** (Vercel serverless):
- First request after idle: ~1-2s extra (Firebase Admin SDK init + Firestore connection)

**Warm request:**
- Auth verification: ~50-100ms (cached JWKs)
- BM25 retrieval: ~10-30ms (in-memory)
- Gemini first token: ~200-500ms
- Full response streaming: ~3-15s depending on answer length

**Memory:**
- `chunks.json` loaded once at module load: ~2.9MB → JS objects ~12MB
- BM25 index (Maps): ~30-50MB in memory
- Fits comfortably in Vercel's 1024MB Node.js runtime

## Known Limitations

1. **No conversation-level RAG.** Retrieval uses only the LAST user message. Multi-turn queries where context matters (e.g. "and what about companies?") won't retrieve the right chunks.

2. **Streaming can lag ~500ms** on Vercel's first request per cold start.

3. **No rate limiting per user.** A malicious user could burn through the free Gemini quota. See "adding rate limits" as a future task.

4. **BM25 tokenizer is basic.** No morphological analysis, no synonyms. Hebrew queries using colloquial terms may miss formal document terms.

5. **Sidebar always shows all chats.** No search, no folders, no archive. Fine for <50 chats per user.

6. **No message editing / regenerating.** Send-only chat model.

## When Things Break — Debugging Guide

- **Client-side errors:** browser DevTools console (F12)
- **Server-side errors in dev:** terminal running `npm run dev`
- **Server-side errors in prod:** Vercel Dashboard → Deployments → Latest → **Logs** (real-time streaming)
- **Firestore permission errors:** Firebase Console → Firestore → **Usage** tab → shows denied requests
- **Auth issues:** Firebase Console → Authentication → **Users** tab
